# ksh26 build system

## Architecture

Three tools, each doing one thing:

```
just build
  → bootstrap samu     (cc -o build/$HOSTTYPE/bin/samu src/cmd/INIT/samu/*.c)
  → run configure.sh   (probes → build.ninja + test infrastructure)
  → samu -C build/$HOSTTYPE
```

| Layer | Tool | What it does |
|-------|------|--------------|
| Porcelain | `just` | User-facing recipes: build, test, clean, diagnostics, CI checks |
| Configure | `configure.sh` | Probes platform, detects features, generates `build.ninja` + test runner |
| Build engine | `samu` | Executes `build.ninja` — fast, parallel, depfile-aware |

`samu` is a vendored C implementation of ninja (ISC/Apache-2.0/MIT licensed,
from michaelforney/samurai). It compiles in under a second with a plain `cc`
invocation, giving us a zero-dependency bootstrap.

**All build and test commands must run inside `nix develop`** (or use
`nix develop -c just <recipe>` for one-shot invocations). The nix flake
provides the complete toolchain: compiler, linker, pkg-config, utf8proc,
libiconv, and debugging tools.

## Nix integration

The nix flake (`flake.nix`) provides the complete build environment and
CI-equivalent checks.

### Package

`nix build` produces a derivation that bootstraps samu, runs configure.sh,
builds with samu, and installs `bin/ksh` and `bin/shcomp`. Dependencies
(utf8proc, libiconv on Darwin) are declared as `buildInputs`.

### Devshells

| Shell | Entry | Purpose |
|-------|-------|---------|
| `default` | `nix develop` | Full build+test environment. stdenv (cc, ld, ar), just, git, pkg-config, ccache, dash. lldb on Darwin; gdb+valgrind on Linux. Dependencies inherited from the package via `inputsFrom`. |
| `agent` | `nix develop .#agent` | Extends default. Sets `CC="ccache cc"`, auto-detects HOSTTYPE and runs `just configure` on entry if no `build.ninja` exists. |

Both shells set a 2 GB per-process memory limit on Linux via `ulimit -v`.

### Checks

| Command | Derivation | What it does |
|---------|------------|--------------|
| `just check` | `checks.$system.default` | Build + full test suite in nix sandbox. Asserts ≥114 test stamps. |
| `just check-asan` | `checks.$system.asan` | Builds base + asan variant, runs asan tests in sandbox. |
| `just check-all` | `nix flake check` | All check derivations. |

The check derivations are the CI source of truth. `just check` runs the
same derivation locally.

## How configure.sh works

configure.sh is a single-pass script (~1680 lines) that runs entirely before
compilation. It takes optional flags: `--force` (rerun all probes),
`--debug` (O0, full debug info), `--asan` (AddressSanitizer + UBSan).

The main function runs six phases in order:

### Phase 0: Install public headers

Copies source headers from `src/lib/libast/include/`, `src/lib/libast/std/`,
`src/lib/libcmd/`, and `src/cmd/ksh26/include/` into the build include
directory so all compilation targets and feature probes can find them.

### Phase 1: Feature detection

Two stages:

- **Compiler probe** — inline in configure.sh (~30 lines). Discovers AR
  path, optimization flag (`-Os` or `-O`), and `-fno-strict-aliasing`
  support via direct compile tests. Replaces the legacy 3,200-line
  mamprobe.sh + C+probe + make.probe chain.

- **iffe.sh** ("if feature exists") — compiles small test programs to
  detect platform capabilities. Each test produces a `FEATURE/<name>`
  header with `#define`s. Retained from the AT&T build system.

Feature tests run for each library in dependency order. libcmd and ksh26
run in parallel (both depend only on libast headers being installed):

1. **libast** (42 tests) — standards, common, lib, map, then the rest
2. **libcmd** (4 tests) — symlink, sockets, ids, utsname
3. **ksh26** (8 tests) — math, time, options, fchdir, locale, cmds, poll, rlimits

**Standalone C probes**: Several ksh26 tests originally used AST library
functions in their `output{}` blocks. We probe those values with standalone
C programs instead, removing the circular dependency on libast and allowing
all probing to happen before compilation.

**Caching**: Each iffe output is individually cached. Probes are skipped
when their output file is fresher than both the input file and configure.sh.
A global cache key (`$CC $CFLAGS`) invalidates everything when the compiler
or flags change. With warm cache, reconfigure takes ~5 seconds.

### Phase 2: Library detection

Probes for external libraries that may not be in the default search path:

- **iconv** — tries bare link, then `-liconv`, then searches Nix store
  and common prefixes. Falls back to AST's internal iconv if not found.

- **utf8proc** — tries `-lutf8proc`, then pkg-config, then fetches and
  builds from source (git clone + single-file compile). Optional: the
  line editor works without it but lacks correct Unicode grapheme/width
  support.

Both results are cached and invalidated when configure.sh changes.

**dsymutil gotcha**: Clang 21+ runs dsymutil when `-g` is set, and dsymutil
can't process `/dev/null`. All library probes compile to a temp file
(`$_probe_out`) instead of `-o /dev/null`.

### Phase 3: Generate derived headers

- **shopt.h** — compile-time shell options from `SHOPT.sh` + runtime probes
  (e.g. `/dev/fd` availability, `test -l` support)
- **git.h** — embedded git commit hash
- **cmdext.h / cmdlist.h** — command function prototypes and list macros,
  generated by scanning `b_*` function signatures in libcmd source files
- **conftab.h / conftab.c** — POSIX conf tables, from `conf.tab` + `conf.sh`
- **lc.h / lctab.c** — locale tables, from `lc.tab` via `lcgen`

### Phase 4: Emit build.ninja

Generates a ninja build file with three rule types:

```ninja
rule cc       # compile .c → .o with automatic depfile tracking
rule ar       # archive .o files → .a
rule link     # link .o + .a → executable
```

All source and include paths are absolute. All output paths (obj/, lib/,
bin/) are relative to the build directory, because samu runs with
`-C build/$HOSTTYPE`.

Ninja's depfile mechanism (`-MD -MF $out.d`) automatically tracks which
headers each .c file includes. No manual header dependency declarations —
the first build compiles everything, incremental builds recompile only
what changed.

Source file discovery is done by `collect_*` functions that walk source
directories. libcmd uses an explicit list (11 files — the static builtin
set plus support code); libast and ksh26 use `find`.

### Phase 5: Test infrastructure

Generates two files that let ninja run tests in parallel:

- **test-env.sh** — exports `SHOPT_*` variables derived from `SHOPT.sh`,
  so tests can check which compile-time features are enabled.

- **run-test.sh** — single-test runner invoked by ninja's `test` rule.
  Uses `dash` (not the ksh under test) as the interpreter. Handles:
  - Per-test temp directory (cleaned on exit)
  - Locale setup (C or C.UTF-8)
  - Memory limits (ulimit on Linux, RSS polling on Darwin)
  - 60-second timeout via GNU `timeout`
  - Exit code classification (PASS/FAIL/SEGV/ABRT/TIME/KILL)
  - Per-test log files (kept only on failure)
  - Summary file for the just porcelain to aggregate

Tests are discovered automatically — drop a `.sh` file in
`tests/shell/`, reconfigure, done. Each test gets both C and
C.UTF-8 locale variants (except `locale.sh`, which sets its own).
Time-sensitive tests (`options`, `sigchld`, `subshell`) run in a serial
pool to avoid interference.

## Build variants

configure.sh flags compose to produce separate build directories:

| Flag | Suffix | Effect |
|------|--------|--------|
| (none) | `$HOSTTYPE` | Default: debug info + optimization |
| `--debug` | `$HOSTTYPE-debug` | `-O0` for reliable single-stepping |
| `--asan` | `$HOSTTYPE-asan` | AddressSanitizer + UBSan |
| `--debug --asan` | `$HOSTTYPE-debug-asan` | Both |

Debug info (`-g`) is always included in all variants — zero runtime cost,
enables useful backtraces. Variant builds share feature probes from the
base build via symlinks, so only the C compilation differs.

The justfile provides matching recipes: `build`/`build-debug`/`build-asan`,
`test`/`test-debug`/`test-asan`, `clean`/`clean-debug`/`clean-asan`.

## Test infrastructure

The justfile `test` recipe builds, runs all tests via samu, then prints
a summary with regression detection:

```
FAIL sigchld.C (exit 1, 2 errors)
---
114/115 pass (12s)
vs previous: 1 regressed
```

Regression detection compares the current `summary.log` against
`summary.prev` (saved automatically). It reports tests that changed from
PASS to non-PASS (regressed) or vice versa (improved).

### Diagnostic recipes

| Recipe | Purpose |
|--------|---------|
| `just errors` | Grep build log for errors |
| `just warnings` | Grep build log for warnings |
| `just failures` | Show failed tests + their individual logs |
| `just log [build\|test] [name]` | Show build/test logs |
| `just test-one NAME [LOCALE]` | Run a single test |
| `just test-repeat NAME [N]` | Run a test N times for flakiness |
| `just debug NAME` | Run a test under lldb (Darwin) or gdb (Linux) |

## Library dependency graph

```
ksh (bin/ksh)
└── libshell.a (65 .c files from ksh26/)
    ├── libcmd.a (11 .c files)
    ├── libast.a (~316 .c files)
    └── libdll.a
```

Link order: `-lshell -lcmd -last -ldll -lm` plus detected flags
(`$ICONV_FLAGS`, `$UTF8PROC_LIBS`).

## AST stdio redirection

AST replaces the C standard library's `FILE` type with its own `Sfio_t`
(from the sfio "safe/fast IO" library). This is implemented via:

1. `FEATURE/stdio` header — `#define FILE _sfio_FILE` plus
   `#define fopen _ast_fopen`, etc.
2. `std/` wrapper headers — `std/stdio.h`, `std/wchar.h` that `#include`
   AST equivalents

For this to work, `build/$HOSTTYPE/include/std/` must appear on the include
path *before* system headers, so `#include <stdio.h>` finds AST's wrapper
instead of the system's. This is the `ast_std` include flag in build.ninja.

**For libast itself**: source subdirectory include paths pick up wrappers
from the source tree's `src/lib/libast/std/`.

**For everything else**: the installed copies at `build/$HOSTTYPE/include/std/`
are used, via the `$ast_std` cflags variable.

This is the single biggest coupling point in the codebase.

## macOS-specific concerns

**Case-insensitive filesystem**: APFS (default on macOS) is case-insensitive,
which means `FEATURE/locale` and `features/locale` (the iffe input directory)
collide. The build handles this by excluding `features/` from libast's
subdirectory include path.

**ksh26 vs libast FEATURE namespace**: Both produce `FEATURE/locale` and
`FEATURE/time` with different content. ksh26's copies stay in `ksh26_work/`
and are found via `-I$BUILDDIR/ksh26_work`. They are not copied to the
shared `FEATDIR`.

**Memory limits**: Darwin ignores `RLIMIT_AS` (Apple removed it), so the
test runner polls child RSS via `ps` and kills on excess instead of using
`ulimit -v`.

## Environment variable interface

| Variable | Default | Effect |
|----------|---------|--------|
| `CC` | `cc` | C compiler |
| `CFLAGS` | (from compiler probe) | Compiler flags (merged with probe results) |
| `HOSTTYPE` | (auto-detected) | Target platform (`os.arch-bits`, e.g. `darwin.arm64-64`) |
| `PACKAGEROOT` | (script directory) | Source tree root |

These are the integration points for nix, cross-compilation, CI, etc.
The nix flake sets `CC`, `CFLAGS`, and `HOSTTYPE` to control the build
without touching configure.sh.

## Retained upstream tools

One tool from the original AT&T build infrastructure is still used for
platform probing. The rest (Mamfiles, mamake, mamprobe, bin/package) is retired.

| Tool | Role |
|------|------|
| `iffe.sh` | "If feature exists" — 98 KB battle-tested feature prober. Compiles small test programs to detect platform capabilities. Regression tests in `tests/infra/iffe.sh`. |
| `bin/shtests` | Test harness for serial test runs (`just test-serial`). |

## Adding or removing a library

To add a new library `libfoo`:

1. Add `collect_libfoo_sources()` function to configure.sh
2. Add feature test function if needed (`run_libfoo_features`)
3. Add a ninja section in `emit_ninja()` (cflags, object loop, archive target)
4. Add to ksh's link line if needed

To remove a library (e.g. replacing it with a system library):

1. Remove its `collect_*`, `run_*_features`, and `emit_ninja` section
2. Replace the archive reference in the link line with `-lfoo` or
   pkg-config flags
3. Remove any generated headers it provided

The build is structured so each library is a self-contained section in
both configure.sh and the generated build.ninja.
